# Link Pilot V1 任务拆分（基于 PRD 逻辑 Review）

本文档基于 `docs/link-pilot-prd-v1.md` 的当前版本进行逻辑复核后整理，目标是把 V1 拆成可执行、可验收、依赖清晰的开发任务。

## 0. Review 结论

整体判断：**PRD 主链路成立，可以进入任务拆分和开发准备**。

当前最合理的 V1 主闭环是：

1. 在 Ahrefs 类结果页识别并收集最近 10 / 20 条线索
2. 先写入本地缓存，再异步同步到 Google Sheets
3. 基于规则补充 `site_summary / link_type / site_business_types / context_match_*`
4. 按规则把可操作目标转入 `opportunities`
5. 用户在管理页筛选候选目标
6. 打开真实页面后执行自动填充 + 人工确认提交
7. 记录提交结果与模板

但为了避免后续实现跑偏，V1 必须先收口以下 6 个点。

## 1. 必须先收口的范围

### 1.1 收集入口以 Ahrefs 结果页为主

PRD 的主目标是“在 Ahrefs 类结果页收集最近 10 / 20 条”。

因此 V1 先以这个入口为准：

- `Popup` 负责当前页面的一键收集
- `Options` 负责查看、筛选、处理结果
- “手动输入 URL / 批量导入文件后直接发起采集”不作为 V1 P0 必做能力

结论：`10.1.1 收集首页` 中的手输 URL / 批量导入更适合作为 **V1.1 或 V2 backlog**，不进入当前核心任务链。

### 1.2 收集方案优先采用 DOM 解析，不默认以 API 拦截为主

PRD 明确写的是“插件识别结果页 DOM 并抓取最近展示的 10~20 条结果”。

因此 V1 的实现优先级应是：

- P0：DOM 检测 + DOM 解析
- 备选：若后续验证发现 DOM 不稳定，再评估 API 拦截

结论：现有任务清单里“Ahrefs API 拦截”不应作为默认主方案。

### 1.3 识别增强先走规则引擎，不引入 LLM 作为主依赖

从 MVP 复杂度和稳定性看，V1 的识别增强应以规则为主：

- 页面 URL / 域名特征
- anchor / title 关键词
- 常见页面结构特征
- 简单评分规则

结论：LLM 可以留作未来增强，不进入 V1 主链路。

### 1.4 必须定义“从 collected_backlinks 到 opportunities”的准入规则

这是当前 PRD 里最重要但还不够落地的一段逻辑。

V1 至少要明确：

- 哪些 `link_type` 才允许入池
- `context_match_score` 的最低阈值是多少
- 哪些页面只保留为线索，不进入待提交区
- 人工改判后是否允许强制入池

### 1.5 模板粒度不能只按 domain，至少要支持 path pattern

同域名下经常存在多种表单结构，因此模板设计至少要支持：

- `domain`
- `page_type`
- `path_pattern`

否则“命中模板但填错表单”的概率会很高。

### 1.6 状态流转、去重规则、失败恢复要先定义

V1 若不先定义状态机，后续列表筛选和同步重试会很混乱。

至少要先明确：

- `collected_backlinks` 的状态流转
- `opportunities` 的状态流转
- `submissions` 的状态流转
- 去重主键规则
- sync queue 的重试策略

---

## 2. V1 开发原则

- **先跑通收集链路，再做提交链路**
- **先本地可用，再做远端同步**
- **先规则识别，再做智能增强**
- **先自动填充，再做自动提交**
- **默认人工确认，不承诺全自动提交**

---

## 3. 任务总览

共拆分为 13 个任务，分 4 个阶段推进。

### 阶段 A：规格收口与基础设施（P0）

- 任务 1：冻结 V1 范围与状态机
- 任务 2：定义共享数据模型与仓储接口
- 任务 3：实现本地存储与同步队列
- 任务 4：实现 Google Sheets / GAS 同步链路

### 阶段 B：收集与识别（P0）

- 任务 5：实现 Ahrefs 页面检测与 DOM 收集器
- 任务 6：实现 Popup 收集流程
- 任务 7：实现线索识别与评分引擎

### 阶段 C：机会池与管理页（P1）

- 任务 8：实现机会池转换规则
- 任务 9：实现 Options 列表与待提交区
- 任务 10：实现目标页打开与处理上下文传递

### 阶段 D：提交辅助闭环（P2）

- 任务 11：实现评论页识别与自动填充
- 任务 12：实现模板学习与提交记录
- 任务 13：实现批量处理、重试与稳定性验证

## 4. 详细任务拆分

### 任务 1：冻结 V1 范围与状态机

**优先级**：P0  
**依赖**：无

#### 目标

把 PRD 中容易引发偏差的边界先固化，避免开发阶段反复返工。

#### 产出

- V1 范围清单
- 明确不做清单
- 状态机定义
- 去重规则
- 机会池准入规则

#### 必须明确的决策

- V1 的唯一收集入口是否就是 Ahrefs 类页面
- `blog_comment` 是否是唯一进入提交闭环的类型
- `context_match_score` 的阈值如何定义
- 什么情况下允许人工强制转成 `opportunity`
- 哪些字段必须同步到 Sheets，哪些仅保留本地

#### 验收标准

- [ ] 形成一版冻结后的 V1 边界说明
- [ ] 三张核心状态机可直接用于开发
- [ ] 去重规则与入池规则已明确

---

### 任务 2：定义共享数据模型与仓储接口

**优先级**：P0  
**依赖**：任务 1

#### 目标

在代码层建立统一的数据契约，避免 popup、options、content、storage 各自维护一套字段。

#### 产出

- `CollectedBacklink` 类型
- `Opportunity` 类型
- `SiteTemplate` 类型
- `Submission` 类型
- `SyncJob` 类型
- 仓储接口与 service 接口

#### 额外要求

- 模板结构补充 `page_type`、`path_pattern`
- `raw_snapshot` 需要限制体积，避免超出 Sheets 单元格承载能力
- 增加 `dedupe_key` 或等效字段

#### 验收标准

- [ ] 所有共享类型已统一
- [ ] 页面层不直接依赖底层存储细节
- [ ] 状态字段和枚举与任务 1 一致

---

### 任务 3：实现本地存储与同步队列

**优先级**：P0  
**依赖**：任务 2

#### 目标

实现 “本地先落盘，再异步同步” 的兜底能力。

#### 范围

- 封装 `chrome.storage.local`
- 本地 CRUD
- sync queue 入队 / 出队 / 重试
- 基础去重
- 批次记录

#### 验收标准

- [ ] 收集结果可稳定写入本地
- [ ] 同步失败不丢数据
- [ ] 重启插件后队列状态可恢复
- [ ] 可按批次查询本次收集结果

---

### 任务 4：实现 Google Sheets / GAS 同步链路

**优先级**：P0  
**依赖**：任务 2、任务 3

#### 目标

建立最小可用的远端持久化链路，但不把复杂认证和管理能力做到过重。

#### 推荐方案

- 使用 Google Apps Script 作为轻量 API 层
- 插件仅调用受控接口
- 支持初始化 4 张 Sheet

#### 必须明确

- 单人使用场景下的鉴权方式
- Sheet 表头初始化方式
- 批量写入节流策略
- 本地状态与远端状态不一致时的回写规则

#### 验收标准

- [ ] 可初始化 `collected_backlinks / opportunities / site_templates / submissions`
- [ ] 本地数据可批量同步到 Sheets
- [ ] 失败任务可标记并重试
- [ ] 插件可展示同步状态

---

### 任务 5：实现 Ahrefs 页面检测与 DOM 收集器

**优先级**：P0  
**依赖**：任务 2、任务 3

#### 目标

稳定识别 Ahrefs 类结果页，并从当前展示结果中抓取最近 10 / 20 条外链线索。

#### 范围

- 页面识别规则
- DOM 解析规则
- 字段映射
- 数据标准化
- 异常容错

#### 注意点

- 只抓“当前页面可获取”的最近结果
- 不在 P0 处理中深分页抓取
- DOM 失效时给出明确错误态

#### 验收标准

- [ ] 能检测当前页是否可收集
- [ ] 能稳定抓取 10 / 20 条结果
- [ ] 抓取字段满足 PRD 8.1
- [ ] 单条解析失败不影响整批保存

---

### 任务 6：实现 Popup 收集流程

**优先级**：P0  
**依赖**：任务 3、任务 5

#### 目标

在用户当前浏览页完成最短路径收集动作。

#### 范围

- 页面状态提示
- “收集最近 10 条 / 20 条”按钮
- 收集批次结果反馈
- 同步状态显示
- 跳转 Options 页面

#### 验收标准

- [ ] 非目标页时给出明确提示
- [ ] 点击后可立即看到收集结果
- [ ] 本地保存与同步状态可见
- [ ] 重复点击不会产生明显重复脏数据

---

### 任务 7：实现线索识别与评分引擎

**优先级**：P0  
**依赖**：任务 2、任务 3、任务 5

#### 目标

对已收集线索进行轻量补充，支撑后续机会池转换。

#### 范围

- `site_summary` 生成
- `link_type` 识别
- `site_business_types` 识别
- `context_match_score` 计算
- `context_match_note` 生成

#### 实现原则

- 先规则引擎
- 不把外部 AI 服务作为主依赖
- 保留人工修正空间

#### 验收标准

- [ ] 每条线索都能得到基础补充字段
- [ ] 无法判断时能明确落到 `unknown`
- [ ] 评分规则可解释、可复现

---

### 任务 8：实现机会池转换规则

**优先级**：P1  
**依赖**：任务 7

#### 目标

把“原始线索”与“可处理机会”真正区分开。

#### 范围

- 自动入池规则
- 人工强制入池
- 忽略 / 归档规则
- `can_submit / can_auto_fill / can_auto_submit` 初始化

#### 建议默认规则

- 仅 `blog_comment` 默认进入提交闭环
- `can_auto_submit` 默认 `false`
- 评分不足或类型不明时仅保留在线索池

#### 验收标准

- [ ] 不会把所有 Ahrefs 线索直接视为待提交目标
- [ ] 入池规则可追踪、可解释
- [ ] 人工改判后状态正确流转

---

### 任务 9：实现 Options 列表与待提交区

**优先级**：P1  
**依赖**：任务 3、任务 7、任务 8

#### 目标

提供统一的管理页，完成查看、筛选、标记和处理入口。

#### 范围

- 已收集线索列表
- 待提交机会区 / Tab
- 类型筛选、状态筛选、搜索
- 批次视图
- 单条打开目标页

#### 验收标准

- [ ] 用户可查看收集结果与补充字段
- [ ] 用户可筛选待提交目标
- [ ] 用户可手动调整状态或备注
- [ ] 用户可从列表直接打开目标页面

---

### 任务 10：实现目标页打开与处理上下文传递

**优先级**：P1  
**依赖**：任务 8、任务 9

#### 目标

让“从待提交区进入真实目标页”这一步具备上下文，不丢失当前待处理记录。

#### 范围

- 打开目标页
- 传递 `opportunity_id`
- 传递预填充数据
- 记录“处理中”状态

#### 验收标准

- [ ] 从 Options 打开页面后，content script 能拿到当前机会上下文
- [ ] 返回列表后仍能看到处理状态
- [ ] 多标签页不会互相串数据

---

### 任务 11：实现评论页识别与自动填充

**优先级**：P2  
**依赖**：任务 10

#### 目标

在 `blog_comment` 页面完成可控的自动识别与自动填充。

#### 范围

- 页面类型识别
- 表单字段识别
- 自动填充
- 手动确认前不自动提交
- 异常提示

#### 验收标准

- [ ] 能识别常见评论表单字段
- [ ] 能自动填充 name / email / website / comment
- [ ] 默认不自动点击提交按钮
- [ ] 识别失败时可进入手动标注模式

---

### 任务 12：实现模板学习与提交记录

**优先级**：P2  
**依赖**：任务 11

#### 目标

把一次成功处理沉淀成可复用模板，并记录提交结果。

#### 范围

- 手动标注字段
- 模板保存 / 复用
- 模板版本更新
- 提交记录落地

#### 验收标准

- [ ] 同站再次访问可优先命中模板
- [ ] 模板支持版本更新与覆盖
- [ ] 成功/失败/未知结果都可记录
- [ ] 提交记录可回看

---

### 任务 13：实现批量处理、重试与稳定性验证

**优先级**：P2  
**依赖**：任务 4、任务 11、任务 12

#### 目标

在不破坏默认安全边界的前提下，补齐 V1 的效率能力和收尾稳定性。

#### 范围

- 批量打开待处理页面
- 自动填充后等待人工确认
- 同步失败重试
- 错误日志与关键提示
- 核心流程测试

#### 注意点

- V1 默认只支持“批量打开 + 自动填充 + 人工确认”
- “自动提交”仅保留受控开关，不作为默认能力

#### 验收标准

- [ ] 可批量打开目标并串行处理
- [ ] 失败任务可暂停、跳过或重试
- [ ] 核心链路至少有基础测试覆盖
- [ ] 不会因为单页异常阻塞整个批次

## 7. 建议开发顺序

如果按最小闭环推进，建议顺序如下：

1. 先做任务 1、2、3，打牢数据与状态基础
2. 接着做任务 5、6，尽快把收集动作跑通
3. 再做任务 4、7、8、9，把“线索 -> 机会”链路补齐
4. 最后做任务 10、11、12、13，完成提交辅助闭环

这个顺序的核心原因是：**先证明“能稳定收集”，再证明“能辅助提交”**。
